# Ручное тестирование MyWay — тест-кейсы по ролям

Документ описывает сценарии **ручной** проверки функций с детализацией до подписей кнопок и полей интерфейса (актуально для текущей русскоязычной UI). Ожидаемые результаты сформулированы так, чтобы их можно было проверить без доступа к коду.

**Обозначения**

- Базовый URL приложения обозначен как `{ORIGIN}` (например `http://localhost:5173`).
- Рабочая область организации: `{ORIGIN}/myway/{slug}/manage/...`, где `{slug}` — адрес студии.
- Тестовые значения ниже — примеры; в вашей среде подставьте реальные сущности (зал, предмет), созданные подготовкой данных.

**Автоматизация (Playwright)**

- Матрица покрытия: [playwright-coverage-matrix.md](playwright-coverage-matrix.md)
- Подготовка данных и запуск: [e2e-seed.md](e2e-seed.md)

**Подготовка тестовых пользователей**

| Роль | Назначение |
|------|------------|
| `SUPER_ADMIN` | Учётка платформы |
| `OWNER` | Владелец тестовой студии |
| `ADMIN` | Администратор той же студии |
| `SUB_TENANT` | Субарендатор |
| `INSTRUCTOR` | Преподаватель (закреплён за предметом — для кейсов с тарифами) |
| `STUDENT` | Ученик (прикреплён к занятию — для посещаемости/пропусков) |

Рекомендуется завести отдельную студию `{slug}` только для QA, чтобы не затронуть прод.

## Матрица доступа к пунктам бокового меню (организация)

Для ролей участников студии меню строится одинаково по пунктам, но **серверные операции** могут вернуть ошибку (например 403), если роли не хватает. Колонка «Меню» — видимость пункта; колонка «Изменение данных» — типичная возможность сохранить результат на backend.

| Пункт меню | Все роли кроме SUPER_ADMIN видят пункт | Кто может менять данные по API/UI |
|------------|----------------------------------------|-------------------------------------|
| Главная | да | все авторизованные в организации |
| Предметы | да | модерация и каталог — OWNER, ADMIN; тарифы — ещё INSTRUCTOR по своим предметам |
| Расписание | да | создание/редактирование — OWNER, ADMIN, INSTRUCTOR; удаление записи — OWNER, ADMIN |
| Залы | да | OWNER, ADMIN |
| Пропуска | да | шаблоны/выдача/журнал админом — OWNER, ADMIN; свои пропуска — все |
| Биллинг | да | OWNER, ADMIN |
| Финансы | да у OWNER при тарифе; у ADMIN — только если OWNER включил флаг и тариф | OWNER (всегда при тарифе); ADMIN — при флаге |
| Субаренда | да | счета/админ — OWNER, ADMIN; бронирования — ещё SUB_TENANT |
| Экспорт 1С | да | OWNER, ADMIN |
| Преподаватели / Ученики | да | приглашение/роли — OWNER, ADMIN |
| График админов | да | OWNER, ADMIN |
| Новости | да | создание/редактирование — OWNER, ADMIN |
| Заявки | да | OWNER, ADMIN |
| Личный кабинет | да | все |
| Настройки | да | см. кейсы (часть полей только OWNER/ADMIN) |
| Обратная связь | да | создание тикета — все |

`SUPER_ADMIN` в контексте `/myway/{slug}/manage` видит только **Главная**, **Платформа**, **Обратная связь**.

## Общие кейсы (без привязки к роли организации)

### TC-GEN-001 — Вход с неверными учётными данными

**Роль:** любая учётная запись не используется (проверка формы).

**Предусловия:** открыт `{ORIGIN}/myway/login`.

**Шаги:**

1. В поле **«Email»** ввести `wrong@example.com`.
2. В поле **«Пароль»** ввести `wrongpass1`.
3. Нажать синюю кнопку **«Войти»**.

**Ожидаемый результат:** вход не выполнен; красное уведомление (toast) с текстом вроде «Неверный пароль или имя пользователя!» или сообщением от сервера; пользователь остаётся на странице входа.

**Как проверить:** URL не меняется на `/myway/.../manage`; в DevTools → Network запрос логина возвращает HTTP 401.

### TC-GEN-002 — Успешный вход OWNER

**Роль:** OWNER.

**Предусловия:** известны email и пароль владельца; студия создана, известен `{slug}`.

**Шаги:**

1. Открыть `{ORIGIN}/myway/login`.
2. **Email:** действующий email владельца.
3. **Пароль:** действующий пароль.
4. Нажать **«Войти»**.

**Ожидаемый результат:** переход на `{ORIGIN}/myway/{slug}/manage/dashboard`; в шапке отображается название организации; боковое меню содержит пункты студии (Главная, Предметы, …).

**Как проверить:** URL совпадает со slug организации пользователя; после обновления страницы сессия сохраняется.

### TC-GEN-003 — Регистрация новой студии (первый владелец)

**Роль:** будет создан пользователь с ролью OWNER новой организации.

**Предусловия:** email не занят; slug свободен.

**Шаги:**

1. Открыть `{ORIGIN}/myway/register`.
2. **«Имя»:** `Тест Регистрация`.
3. **«Email»:** уникальный, например `studio-owner-{timestamp}@example.com`.
4. **«Пароль»:** минимум 8 символов, например `TestPass12`.
5. **«Название студии»:** `Студия Автотест`.
6. **«Адрес студии (URL)»:** уникальный латиницей, например `autotest-{timestamp}` (или оставить автозаполнение из названия и дождаться подсказки «свободен»).
7. Включить чекбокс с текстом про **Политику** и **Пользовательское соглашение** (обязательный).
8. По желанию включить чекбокс маркетинговых сообщений.
9. Нажать **«Зарегистрироваться»**.

**Ожидаемый результат:** успешная регистрация; переход в `/myway/{slug}/manage/dashboard`; пользователь — владелец новой организации.

**Как проверить:** slug в URL совпадает с нормализованным адресом; в настройках организации отображается имя «Студия Автотест».

### TC-GEN-004 — Публичная заявка преподавателя

**Роль:** после одобрения — INSTRUCTOR (до одобрения — ограниченный статус).

**Предусловия:** у студии есть хотя бы один предмет в публичном каталоге или допускаются произвольные названия.

**Шаги:**

1. Открыть `{ORIGIN}/myway/{slug}/join/instructor`.
2. **«Имя»:** `Препод Иван`.
3. **«Email»:** уникальный `instr-{timestamp}@example.com`.
4. **«Пароль для входа в личный кабинет»:** `TestPass12`.
5. **«Предметы из списка студии»:** выбрать один предмет из выпадающего списка ИЛИ в **«Если нужного предмета нет в списке»** ввести тег `Pole Dance` (Enter).
6. **«Сообщение владельцу»:** `Автотест заявки`.
7. Включить обязательный чекбокс согласия с документами.
8. Нажать **«Подать заявку»**.

**Ожидаемый результат:** либо экран **«Заявка отправлена»** (если включено подтверждение заявок), либо **«Регистрация завершена»**; кнопки **«На страницу студии»** / **«Войти»** по сценарию.

**Как проверить:** под OWNER открыть **«Заявки»** — строка с email заявителя; после одобрения вход под этим email открывает manage-кабинет.

### TC-GEN-005 — Публичная заявка ученика с тарифом

**Роль:** STUDENT после одобрения/активации.

**Предусловия:** у выбранного предмета есть утверждённый тарифный план.

**Шаги:**

1. Открыть `{ORIGIN}/myway/{slug}/join/student`.
2. Заполнить **«Имя»**, **«Email»** (уникальный), **«Пароль для входа»** (`TestPass12`).
3. В **«Предметы из списка студии»** выбрать предмет с тарифами.
4. В **«Тарифные планы (опционально)»** выбрать строку вида `Имя предмета · Название тарифа (... ₽/занятие)`.
5. Согласие — чекбокс включён.
6. Нажать **«Записаться»**.

**Ожидаемый результат:** успех по логике настройки «Заявки требуют подтверждения» (как в TC-GEN-004).

**Как проверить:** в списке заявок отображается количество тарифов в колонке **«Тарифы»**.

### TC-SA-002 — Платформа: фильтр организаций и назначение подписки (smoke)

**Предусловия:** открыта `/myway/platform`; есть минимум одна организация.

**Шаги:**

1. В таблице организаций выбрать строку (клик по радио/выбору, если есть — следуйте фактическому UI).
2. Найти блок управления подпиской (форма с планом).
3. Выбрать в поле плана известное значение из списка **«Тариф»** / plan code (например доступный в тестовой среде план).
4. Нажать кнопку сохранения подписки (в интерфейсе — действие подтверждения назначения).

**Ожидаемый результат:** сообщение об успехе; у выбранной организации обновился статус/тариф в таблице после обновления данных.

### TC-SA-025 — SMTP: исходящая почта с ящика админа (рассылка платформы)

**Тип:** ручной (без Playwright; проверка доставки на реальный почтовый ящик).

**Предусловия:**

- На стенде (пилот/prod) в `~/.myway/compose.env` настроен **реальный SMTP** (`MAIL_HOST` и др.).
- Пример (Timeweb): `MAIL_HOST=smtp.timeweb.ru`, `MAIL_PORT=465`, `MAIL_USERNAME=admin@myway-line.ru`, `MAIL_PASSWORD=<из панели>`, `MAIL_SMTP_SSL=true`, `PLATFORM_EMAIL_FROM=admin@myway-line.ru`.
- После правок `.env` выполнен перезапуск backend: `docker compose --env-file .env -f docker-compose.prod.yml up -d --force-recreate backend`.
- Учётка **SUPER_ADMIN** или **SUPER_USER**.
- Есть тестовая студия, у которой **OWNER** с **реальным email** (ваш личный ящик для приёма), подписка на тариф с известным **кодом плана** (например `TESTER`, `DEVELOPER`).

**Шаги:**

1. Войти как SUPER_ADMIN → `{ORIGIN}/go/platform` → вкладка **«Рассылки»**.
2. В карточке **«Новая рассылка (черновик)»** заполнить:
   - **Название:** `TC-SA-025 SMTP test`;
   - **Тема письма:** `TC-SA-025 Проверка ящика admin`;
   - **Текст:** `Проверка исходящей почты MyWay. Студия: {orgName}.`;
   - **Тарифы (коды):** тег с кодом плана тестовой студии (тот, на котором сидит OWNER из предусловий).
3. Нажать **«Создать черновик»** — в таблице кампаний появилась строка со статусом черновика и **числом получателей ≥ 1**.
4. Нажать **«Отправить»** у этой кампании и подтвердить отправку.
5. Открыть почтовый ящик OWNER (из предусловий). При необходимости проверить папку **«Спам»**.

**Ожидаемый результат:**

- В UI кампания переходит в статус отправленной; по каналу **EMAIL** у получателя — **`SENT`** (не `FAILED`, не `SKIPPED_NO_OWNER_EMAIL`).
- На почте OWNER пришло письмо:
  - **От:** `PLATFORM_EMAIL_FROM` (например `admin@myway-line.ru`);
  - **Тема:** содержит `TC-SA-025` / префикс платформы;
  - в теле подставлено имя студии вместо `{orgName}`.
- Во **входящих** OWNER на **«Главная»** студии появилось in-app сообщение с той же темой (канал IN_APP).

**Как проверить при сбое:**

- `docker compose -f docker-compose.prod.yml logs backend --tail 200 | grep -i mail` — нет `Platform message email failed` / `Failed to send verification email`.
- Сверить логин/пароль SMTP в панели хостинга почты с `MAIL_USERNAME` / `MAIL_PASSWORD` в `.env`.

**Связанные кейсы:** PLT-06 (сообщение одному владельцу), NFT-05 (общий smoke SMTP), `deploy/pilot/PILOT-STAND-MEMO.md` (§ Почта).

**Ожидаемый результат:** удаление аккаунта недоступно (кнопка отключена или предупреждение).

---

## Регресс и быстрый чек-лист по ролям (smoke)

| Роль | Минимальный smoke за 10 минут |
|------|--------------------------------|
| SUPER_ADMIN | вход → platform → рассылка TC-SA-025 (SMTP) или сообщение владельцу (PLT-06) |
| OWNER | вход → настройки организации → создать зал → создать занятие → открыть финансы |
| ADMIN | вход → расписание → заявки → проверить финансы согласно флагу |
| SUB_TENANT | вход → субаренда → негатив на экспорт |
| INSTRUCTOR | вход → новая запись → посещаемость → тариф на согласовании |
| STUDENT | вход → расписание read-only → ЛК → QR |
