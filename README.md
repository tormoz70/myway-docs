# MyWay — Сервис управления арендой студий

Веб-приложение для управления бронированием и арендой студий: расписание, клиенты, оплата и аналитика в одном месте.

## Стек технологий

| Слой | Технологии |
|------|-----------|
| Backend | Java 21, Spring Boot 3, Gradle |
| Frontend | React, TypeScript, Vite |
| БД | PostgreSQL 16 |
| Кэш | Redis 7 |
| Хранилище файлов | MinIO (S3-совместимое) |
| CI/CD | GitHub Actions |
| Контейнеризация | Docker, Docker Compose |

## Быстрый старт

### Запуск инфраструктуры

```bash
cd backend
docker-compose up -d
```

Будут подняты:
- **PostgreSQL** — `localhost:5432` (db=myway, user=myway, password=myway)
- **Redis** — `localhost:6379`
- **MinIO** — API `localhost:9000`, Консоль `localhost:9001` (minioadmin / minioadmin)

### Запуск бэкенда

```bash
cd backend
./gradlew bootRun
```

Сервер будет доступен на `http://localhost:8080`.

### Запуск фронтенда

```bash
cd frontend
npm install
npm run dev
```

Приложение будет доступно на `http://localhost:5173`.

## Роли и права доступа (RBAC)

В системе используются роли:
- `SUPER_ADMIN` — суперадмин платформы
- `OWNER` — владелец организации
- `ADMIN` — администратор
- `SUB_TENANT` — арендатор
- `INSTRUCTOR` — преподаватель
- `STUDENT` — ученик

### Матрица доступа

| Роль | Основное назначение | Ключевые права |
|------|----------------------|----------------|
| `SUPER_ADMIN` | Управление всей платформой | Просмотр всех организаций и их владельцев, управление тарифами/подписками/платежами платформы, обработка обратной связи |
| `OWNER` | Полное управление организацией | Все права `ADMIN`, управление участниками организации (приглашение, редактирование профиля/роли, удаление) |
| `ADMIN` | Операционное управление сервисом | Управление залами, расписанием, пропусками, биллингом, субарендой, счетами и экспортом |
| `SUB_TENANT` | Работа с арендой | Создание/редактирование/отмена/восстановление бронирований субаренды без прав подтверждения и выставления счетов |
| `INSTRUCTOR` | Проведение занятий | Работа с расписанием (создание/редактирование), операции check-in |
| `STUDENT` | Участник занятий | Операции check-in и доступ к разрешенным просмотрам для авторизованных пользователей |

### Ограничения по чувствительным операциям

- Подтверждение/снятие подтверждения бронирования: только `OWNER`/`ADMIN`
- Выставление счетов и регистрация оплат: только `OWNER`/`ADMIN`
- Управление пользователями организации: только `OWNER`/`ADMIN`
- Экспорт данных: только `OWNER`/`ADMIN`
- Платформенные операции (тенанты, подписки, обращения): только `SUPER_ADMIN`

### Примечания

- Проверка принадлежности к организации выполняется на backend (organization-scoped access).
- У владельца (`OWNER`) есть защита от случайной деградации прав: роль владельца нельзя понизить и владельца нельзя удалить через API управления участниками.
- Роль `SUPER_ADMIN` предназначена для платформенного контура и не назначается через обычный интерфейс управления участниками организации.

## Платформенные подписки

Подписочная модель разделена на планы платформы и подписки организаций:
- планы: `DEVELOPER`, `TEST_OWNER_FREE`, `TESTER`, `START`, `PRO`, `ENTERPRISE`;
- назначение/смена плана выполняется `SUPER_ADMIN` в разделе платформы;
- для `TEST_OWNER_FREE` по умолчанию отключено создание коммерческих счетов до перевода на коммерческий план.

### Подробное описание тарифов

#### 1) `DEVELOPER`
- Назначение: внутренний план для разработки и техподдержки.
- Стоимость: `0 ₽/мес`, `0 ₽/год`.
- Лимиты:
  - комнаты: до `50`;
  - пользователи: до `200`;
  - check-in в месяц: до `100000`;
  - бронирования в месяц: до `10000`;
  - экспорт 1С: включен.
- Срок: длительный сервисный режим (`defaultDurationDays=3650`).

#### 2) `TEST_OWNER_FREE`
- Назначение: бесплатный тестовый план для владельца тестового тенанта.
- Стоимость: `0 ₽/мес`, `0 ₽/год`.
- Лимиты:
  - комнаты: до `3`;
  - пользователи: до `10`;
  - check-in в месяц: до `200`;
  - бронирования в месяц: до `20`;
  - экспорт 1С: выключен.
- Срок: `60` дней.
- Особенность: коммерческие платформенные счета для этого плана по умолчанию не выставляются.

#### 3) `TESTER`
- Назначение: льготное пользовательское тестирование.
- Стоимость: `0 ₽/мес`, `0 ₽/год`.
- Лимиты:
  - комнаты: до `5`;
  - пользователи: до `20`;
  - check-in в месяц: до `500`;
  - бронирования в месяц: до `50`;
  - экспорт 1С: выключен.
- Срок: `90` дней.

#### 4) `START`
- Назначение: базовый коммерческий тариф.
- Стоимость: `9900 ₽/мес`, `99000 ₽/год`.
- Лимиты:
  - комнаты: до `10`;
  - пользователи: до `50`;
  - check-in в месяц: до `3000`;
  - бронирования в месяц: до `300`;
  - экспорт 1С: включен.
- Срок подписки по умолчанию: `30` дней.

#### 5) `PRO`
- Назначение: расширенный коммерческий тариф для растущих студий и сетей.
- Стоимость: `29900 ₽/мес`, `299000 ₽/год`.
- Лимиты:
  - комнаты: до `30`;
  - пользователи: до `200`;
  - check-in в месяц: до `20000`;
  - бронирования в месяц: до `2000`;
  - экспорт 1С: включен.
- Срок подписки по умолчанию: `30` дней.

#### 6) `ENTERPRISE`
- Назначение: индивидуальные условия для крупных клиентов.
- Стоимость: согласуется индивидуально (`0` в seed-данных как placeholder).
- Лимиты: без фиксированных ограничений в seed-конфигурации (`NULL`, определяется договором).
- Экспорт 1С: включен.
- Срок подписки по умолчанию: `30` дней.

### Общие правила по лимитам и статусам

- Все лимиты применяются на уровне организации (тенанта).
- `SUPER_ADMIN` может вручную:
  - назначить/сменить план;
  - задать `overridePrice`;
  - включить trial-режим;
  - перевести тестового владельца на коммерческий план.
- Рекомендуемый подход к превышениям:
  - сначала предупреждение (soft-warning и уведомление владельца);
  - затем ограничение новых операций записи;
  - при критичных нарушениях или задолженности — блокировка тенанта (`SOFT`/`HARD`) через платформенный контур.
- Базовые статусы подписки:
  - `TRIAL` — пробный период;
  - `ACTIVE` — активная подписка;
  - `EXPIRED` — срок истек;
  - `CANCELED` — отменена.
- Статусы платформенного счета:
  - `DRAFT` — черновик;
  - `SENT` — отправлен;
  - `PAID` — оплачен;
  - `OVERDUE` — просрочен.

В платформенном биллинге поддерживается ручный контур:
- выставление счета по подписке;
- регистрация оплаты;
- статусы счета: `DRAFT`, `SENT`, `PAID`, `OVERDUE`.

## Обратная связь с разработкой

- Любой авторизованный пользователь может создать обращение через раздел `Обратная связь`.
- Обращение сохраняется в системе и доступно `SUPER_ADMIN` в платформенном разделе.
- При создании обращения отправляется email-уведомление команде разработки (если настроен `FEEDBACK_EMAIL_TO`).

## Dashboard SUPER_ADMIN и блокировки тенантов

На главной странице для `SUPER_ADMIN` доступен отдельный dashboard:
- KPI: общее число тенантов, занятый объем БД (Гб), лимит БД из `APP_DB_CAPACITY_GB` и процент утилизации;
- `Top-10` тенантов по размеру данных;
- полный список тенантов с действиями: сообщение владельцу, блокировка (`SOFT`/`HARD`), разблокировка.

Режимы блокировки:
- `HARD`: вход в систему для пользователей тенанта запрещен;
- `SOFT`: вход разрешен, но операции записи (`POST/PUT/PATCH/DELETE`) блокируются на backend;
- для `OWNER` остается доступ к отправке сообщения `SUPER_ADMIN` даже при блокировке.

### Коммуникация OWNER ↔ SUPER_ADMIN

- `SUPER_ADMIN -> OWNER`: сообщение сохраняется in-app и отправляется владельцу по email.
- `OWNER -> SUPER_ADMIN`: сообщение сохраняется in-app и отправляется:
  - на email активных пользователей с ролью `SUPER_ADMIN`;
  - на дополнительный mailing list из `SUPERADMIN_EMAIL_TO`.

Настройки для платформенных уведомлений:
- `APP_DB_CAPACITY_GB` — общий лимит объема БД (Гб);
- `SUPERADMIN_EMAIL_TO` — дополнительные адреса для уведомлений;
- `PLATFORM_EMAIL_FROM` — отправитель платформенных писем.
